# ADR-0005: Author Identity in Support Notes

**Status**: Accepted  
**Date**: 2026-03-09

## Context

The `POST /api/shops/{shopId}/notes` endpoint creates an internal support note. Each note has
an `author` field identifying who wrote the note. The case description explicitly flags this
as unresolved:

> *"author: object — id: string (pending to confirm), name: string (pending to confirm)"*

And lists as an improvement:

> *"Get a way to fetch users from another system or by decoding a JWT token generated by a
> single source of truth (e.g. Auth0, Okta, etc.) to avoid hardcoding user information in the
> support notes."*

Three approaches were considered:

### Option A — Free-form caller-supplied author (chosen for PoC)

The request body contains `{ author: { id: string, name: string } }` and the server stores it
verbatim. No server-side verification of whether the author exists or is who they claim to be.

### Option B — JWT-derived author identity

The upstream gateway issues a signed JWT containing user claims (`sub`, `name`). The service
verifies the JWT and extracts the author identity — the client never sends `author` in the
body.

### Option C — External user directory lookup

The service calls an external Identity Provider (Auth0 / Okta) or internal user service to
resolve a `userId` to a `{ id, name }` before storing the note.

## Decision

**Use Option A (caller-supplied author) for this iteration.**

Rationale:

1. **No auth infrastructure exists yet**: the case description states *"assume internal auth
   is handled upstream; you only receive X-Tenant-Id"*. There is no JWT in the request.
2. **Scope**: implementing JWT verification or an external user lookup is out of scope for this
   PoC and would add significant complexity without demonstrating the core domain behaviour.
3. **Port/adapter isolation applies here too**: the `NoteAuthor` is a value object in the
   domain. How it is populated (from request body, from JWT claims, from external service) is
   an infrastructure concern that can be changed without touching the domain.

## Security concern (raised explicitly)

**This is a known and deliberate vulnerability in the PoC.**

With Option A, any caller can impersonate any author by sending arbitrary `{ id, name }` values.
In an internal tool accessed by authenticated employees this risk is lower than in a public API,
but it is still unacceptable in production for audit and accountability reasons.

## Production evolution path

1. **JWT claims extraction** (recommended first step): the upstream gateway (Auth0/Okta) adds
   user identity as JWT claims. This service adds a `JwtAuthorExtractor` driven adapter that
   reads `sub` and `name` from the verified token. The request body no longer accepts `author`.
2. **Author immutability**: once stored, the author on a note is never updated — notes are
   immutable records of who said what and when. If a user's display name changes, historical
   notes retain the name at the time of writing.
3. **Audit log**: in production, note creation should emit a domain event (`NoteCreated`) that
   is persisted to an append-only audit log with the author's verified identity and timestamp.

## Note schema (this iteration)

```typescript
// Request body
{
  author: { id: string, name: string },  // caller-supplied, not verified
  content: string                         // non-empty string, max 2000 chars
}

// Stored note (domain entity)
{
  id: string,          // UUID generated server-side
  shopId: string,
  tenantId: string,    // scoped from middleware, not from body
  author: { id: string, name: string },
  content: string,
  createdAt: string    // ISO 8601, generated server-side
}
```

Note: `tenantId` is **never** accepted from the request body. It is always taken from the
`X-Tenant-Id` header (validated by middleware). This prevents a caller from creating notes
scoped to a different tenant.

## Consequences

- ✅ Minimal implementation — no external dependencies, no JWT library needed.
- ✅ `NoteAuthor` is a value object; the extraction strategy is replaceable without domain changes.
- ✅ `tenantId` is always server-derived — tenant isolation is maintained even with free-form authors.
- ⚠️ Author identity is unverified — any caller can claim to be any user. **Not acceptable in
  production.** Tracked as a known limitation and documented in the README.
- ⚠️ No user existence validation — a note can be created with a non-existent `author.id`.
